Lesson 11

Cascade Layers (@layer) — Managing Specificity at Scale

You know specificity: #id beats .class beats element. But in real projects with resets, design tokens, component styles, and utility classes, specificity becomes a war. You end up using !important or over-specific selectors to win fights you shouldn't be fighting.

Quick primer: how specificity is calculated

Specificity is a three-part score written as (A, B, C). You calculate it by counting selector parts and placing the count in the right column:

Specificity = (IDs, Classes/Attrs/Pseudo-classes, Elements/Pseudo-elements)
                ↑              ↑                            ↑
           count of #      count of .class        count of element
                           [attr]                  ::before
                           :hover, :not()          ::after
                           :has(), :is()

Examples:

h1                        → 0 IDs, 0 classes, 1 element     = (0, 0, 1)
.card                     → 0 IDs, 1 class,   0 elements    = (0, 1, 0)
.card .title              → 0 IDs, 2 classes, 0 elements    = (0, 2, 0)
#main .card               → 1 ID,  1 class,   0 elements    = (1, 1, 0)
#main .card .title:hover  → 1 ID,  3 classes, 0 elements    = (1, 3, 0)
div.card[data-v]          → 0 IDs, 2 (class+attr), 1 elem   = (0, 2, 1)

Comparison rule: Read left to right. The first column where one is higher wins. Columns don't add up across positions — (1, 0, 0) beats (0, 99, 99) because one ID outranks any number of classes. Think of it like comparing version numbers: 2.0.0 > 1.99.99.

Now — here's the problem layers solve:

@layer introduces a higher-level control: layers are resolved before specificity is even considered. A style in a higher-priority layer always wins, regardless of how specific the selector is in a lower layer.

The mental model

Think of the cascade as resolving in this order (highest priority wins):

1. Transitions (active transitions always win)
2. !important (in reverse layer order — more on this later)
3. Inline styles (style="...")
4. Unlayered styles (any CSS not inside a @layer)
5. Layers (in declared order — LAST declared layer wins)
6. Specificity (only matters WITHIN the same layer)
7. Source order (last rule wins if everything else is equal)

The key insight: Layers sit ABOVE specificity in the cascade. A p { color: red } in a higher-priority layer beats a #main .content p.highlight { color: blue } in a lower-priority layer. Specificity only breaks ties within the same layer.

Declaring and ordering layers

You declare layers and their order up front, then add styles to them:

This box is styled by layers: reset → base → components. The component layer wins (it's declared last)

I'm styled by the "components" layer (highest priority)
<!-- HTML -->
<div class="layer-demo-box card">
  I'm styled by the "components" layer
</div>

/* Declare order: last = highest priority */
@layer reset, base, components, utilities;

/* Add styles to layers */
@layer reset {
  .layer-demo-box { all: unset; display: block; }
}

@layer base {
  .layer-demo-box { background: teal; color: white; }
}

@layer components {
  .layer-demo-box.card { background: blue; color: white; }  /* wins over base */
}

Layer order rule: Layers are prioritised in the order they're declared. Last declared = highest priority. This is the opposite of what you might expect (you might think "first = most important") but it mirrors source order logic: later things override earlier things.

Declaration order vs definition order — they're different

This is a critical distinction. The declaration (the @layer name1, name2, name3; line) sets priority. The definition (where you actually write the styles inside @layer name { ... }) can be anywhere — in any order, in any file. It doesn't affect priority.

Declaration says: reset < base < utilities. The definitions below are in reverse order — doesn't matter. Utilities still wins.

utilities layer wins (declared last) even though its styles are defined first in the file
/* This line locks in the priority order */
@layer reset, base, utilities;

/* These can appear in ANY order — priority is already set above */
@layer utilities { .box { background: coral; } }
@layer reset { .box { all: unset; display: block; } }
@layer base { .box { background: teal; } }

/* Result: utilities wins (coral) because it was DECLARED last,
   even though its styles are DEFINED first in the file. */

Why this matters: It decouples architecture from authoring. You declare the priority structure once at the top of your stylesheet (or entry file), then teams, files, and imports can add styles to layers in any order without accidentally changing what wins. The declaration is the single source of truth for priority.

Unlayered styles beat all layers

Any CSS that is NOT inside a @layer is considered "unlayered" — and unlayered styles always beat layered styles, regardless of layer order or specificity:

This box has both a "components" layer style AND an unlayered style. Unlayered wins (purple)

Unlayered style wins over all layers
<!-- HTML -->
<div class="layer-demo-box card override-demo">
  Unlayered style wins over all layers
</div>

@layer components {
  .layer-demo-box.card { background: blue; }  /* in a layer */
}

/* This is NOT in any layer — it wins */
.layer-demo-box.override-demo { background: purple; }

Strategy: Put all your CSS into layers. Keep the unlayered space reserved for one-off overrides or debugging. If everything is layered, you control the cascade entirely through layer order.

A practical layer architecture

Here's a common pattern for organising a project:

Layer stack (lowest → highest priority)

1. reset — normalize/reset styles
2. base — element defaults (body, h1-h6, a, p)
3. layout — page structure (grid, sidebar, header/footer)
4. components — reusable components (cards, buttons, forms)
5. utilities — utility/override classes (.hidden, .text-center)
/* Declare the layer order at the top of your stylesheet */
@layer reset, base, layout, components, utilities;

/* Then populate each layer */
@layer reset { /* normalize styles */ }
@layer base { /* element defaults */ }
@layer layout { /* page structure */ }
@layer components { /* .card, .button, .form-field */ }
@layer utilities { /* .hidden, .sr-only, .text-center */ }

Now utilities always beat components, components always beat base — regardless of selector specificity. You never need !important to make a utility class override a component style.

!important and layers — the inversion

Here's the counterintuitive part. For normal styles, later layers win. For !important styles, it's reversed — earlier layers win:

Normal:      reset < base < components < utilities    (utilities wins)
!important:  reset > base > components > utilities    (reset wins!)

Why? Because !important in a reset is meant to be an absolute guarantee ("this element MUST have this style"). If a higher layer's !important could override it, the reset's guarantees would be meaningless.

Mental model: !important inverts layer priority. The "lowest" layer's !important becomes the hardest to override. This means: put your most critical, non-negotiable styles (like accessibility resets) in the lowest layer with !important, and nothing can touch them.

Adding third-party CSS to a layer

You can wrap an imported stylesheet in a layer, giving you control over its priority relative to your own code:

Third-party CSS contained in a low-priority layer — your styles always win

@import url("bootstrap.css") layer(vendor); — Bootstrap is now in a named layer you control 
<!-- HTML -->
<button class="btn btn-primary">Click me</button>

/* Put third-party CSS into a low-priority layer */
@layer vendor, reset, base, components, utilities;

@import url("normalize.css") layer(vendor);
@import url("some-library.css") layer(vendor);

/* Your code in higher layers always overrides vendor styles
   without needing higher specificity or !important */
@layer components {
  .btn { /* your styles beat the library's .btn */ }
}

This solves the classic problem of fighting third-party CSS specificity. Instead of writing .my-app .container .button.primary to beat Bootstrap's selector, you just put Bootstrap in a lower layer.

Nested layers

Layers can be nested. Inner layers are scoped to their parent:

Nested layers for organising component sub-styles

components.buttons < components.cards < components.forms 
<!-- HTML -->
<button class="btn">Default</button>
<button class="btn btn-primary">Primary</button>
<div class="card">A card</div>
<input class="input" type="text">

@layer components {
  @layer buttons {
    .btn { padding: 0.5rem 1rem; }
  }

  @layer cards {
    .card { border-radius: 8px; }
  }

  @layer forms {
    .input { border: 1px solid gray; }
  }
}

/* Or reference nested layers with dot notation: */
@layer components.buttons {
  .btn-primary { background: blue; }
}

Anonymous layers

You can create layers without naming them. They get a unique position in the order but can't be appended to later:

/* Anonymous layers — useful for one-off isolation */
@layer {
  .widget { /* these styles are layered but unnamed */ }
}

@layer {
  .other { /* separate anonymous layer — different priority */ }
}

Use named layers in most cases. Anonymous layers are useful for completely isolating a block of CSS (like a third-party widget embed) without giving it a name that could conflict.

How this changes your workflow

Before layers, managing specificity meant:

  • BEM naming to keep specificity flat
  • Fighting third-party libraries with over-specific selectors
  • !important escalation wars
  • Source order dependency (CSS file loading order matters)

With layers:

  • Declare the priority order once at the top
  • Never worry about selector specificity across layers
  • Third-party CSS goes in a low layer — your code always wins
  • Source order of layer content doesn't matter — only the layer declaration order matters

Browser support

@layer is supported in Chrome 99+, Safari 15.4+, Firefox 97+. Fully production-ready.
