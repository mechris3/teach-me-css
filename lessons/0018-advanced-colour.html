Lesson 18

Advanced Colour — OKLCH, Wide Gamut & Relative Colour Syntax

For 25 years, web colour meant hex (#ff6600), rgb(), and maybe hsl(). These all describe colours within the sRGB gamut — a limited range of colours defined in 1996. Modern displays (P3 screens on MacBooks, iPhones, etc.) can show about 50% more colours than sRGB can represent. And perceptually, sRGB tools are broken — "50% lightness" in HSL doesn't look half as bright to your eyes.

Modern CSS fixes both problems with new colour spaces and syntax.

The mental model: why oklch matters

OKLCH has three channels:

  • L (Lightness) — 0% = black, 100% = white. Perceptually uniform: L=50 genuinely looks "half-bright" to human eyes.
  • C (Chroma) — how saturated/vivid the colour is. 0 = grey, higher = more vivid. No fixed max (depends on the hue).
  • H (Hue) — the colour angle on the colour wheel (0–360°). 0=red, 90=yellow, 180=cyan, 270=blue.

Because hue is an angle, it wraps: 370° = 10°. And you can use calc(h + 180) in relative colour syntax to get the complementary colour — the one directly opposite on the wheel.

Why not just HSL? HSL lies to you. hsl(60, 100%, 50%) (yellow) looks FAR brighter than hsl(240, 100%, 50%) (blue) — even though both claim "50% lightness." That's because HSL's lightness is a mathematical average of RGB channels, not a perceptual measurement. OKLCH's lightness is calibrated to human vision — same L value = same perceived brightness regardless of hue.

oklch() syntax

Same lightness (65%), same chroma (0.2), different hues — they look equally bright

<!-- HTML -->
<div class="swatch" style="background: oklch(65% 0.2 0);">red</div>
<div class="swatch" style="background: oklch(65% 0.2 120);">green</div>
<div class="swatch" style="background: oklch(65% 0.2 240);">blue</div>

/* oklch syntax */
color: oklch(65% 0.2 240);
/*           L    C    H
             │    │    └── Hue angle (0–360°)
             │    └─────── Chroma (0 = grey, ~0.4 = max vivid)
             └──────────── Lightness (0% = black, 100% = white)
*/

/* With alpha (transparency) */
color: oklch(65% 0.2 240 / 0.5);  /* 50% transparent */

color: oklch(65% 0.25 265);  /* blue — same perceived brightness */

This is why OKLCH matters for design systems: You can create a palette where all colours at the same lightness level are actually interchangeable as backgrounds without adjusting text contrast. In HSL, you'd need manual tweaks per hue.

Wide gamut — colours beyond sRGB

sRGB can only represent a subset of colours visible to the human eye. Modern displays (Apple P3, Rec.2020) can show more. OKLCH and color() can express these wider-gamut colours:

On a P3 display, the top row (P3 gamut) will appear more vivid than the bottom (sRGB). On sRGB displays, they'll look the same.

/* Wide gamut — more vivid on P3 displays */
color: oklch(65% 0.3 25);      /* a red more vivid than #ff0000 can express */
color: color(display-p3 1 0 0); /* pure P3 red — outside sRGB entirely */

/* The color() function for specific colour spaces */
color: color(display-p3 0.2 0.8 0.35);  /* P3 green */
color: color(srgb 0.2 0.5 1.0);          /* explicit sRGB */

/* Fallback for browsers/displays without wide gamut */
.vibrant {
  color: rgb(255, 59, 48);              /* sRGB fallback */
  color: oklch(65% 0.3 25);            /* wide gamut (used if supported) */
}

Graceful degradation: If a browser or display can't show a wide-gamut colour, it gamut maps — clamps it to the closest representable colour in the display's gamut. The page doesn't break; it just looks slightly less vivid. You can safely use oklch everywhere.

Relative colour syntax — derive colours from other colours

This is where it gets powerful. Relative colour syntax lets you take any colour and create variations by modifying individual channels:

One base colour → an entire tint/shade scale by modifying lightness

L: 95%
L: 85%
L: 75%
L: 65%
L: 50%
L: 35%
L: 20%
/* Relative colour syntax: oklch(from <origin> L C H) */
/* "from" converts the origin colour into oklch, then you override channels */

:root {
  --brand: #3a86ff;
}

/* Lighten — keep chroma and hue, increase lightness */
--brand-100: oklch(from var(--brand) 95% c h);
--brand-200: oklch(from var(--brand) 85% c h);
--brand-300: oklch(from var(--brand) 75% c h);

/* Darken — decrease lightness */
--brand-700: oklch(from var(--brand) 35% c h);
--brand-900: oklch(from var(--brand) 20% c h);

/* Desaturate — reduce chroma */
--brand-muted: oklch(from var(--brand) l 0.05 h);

/* Shift hue — create complementary/analogous */
--complement: oklch(from var(--brand) l c calc(h + 180));
--analogous:  oklch(from var(--brand) l c calc(h + 30));

The syntax explained

oklch(from <origin-color> <L> <C> <H> / <alpha>)

/* "from" converts the origin into oklch channels: l, c, h */
/* You can then use those channel names in expressions: */

oklch(from red l c h)              /* = red (no changes) */
oklch(from red 80% c h)           /* = lighter red (L overridden to 80%) */
oklch(from red l 0.1 h)           /* = desaturated red (C reduced) */
oklch(from red l c calc(h + 90))  /* = hue-shifted 90° (red → yellow-ish) */
oklch(from red l c h / 0.5)       /* = 50% transparent red */

The power here: You define ONE brand colour, and derive your entire palette mathematically. Change the brand colour → the whole palette updates. No manual tuning per shade. And because it's in oklch, the lightness steps look perceptually even.

Relative colour syntax with other spaces

It works with any colour function, not just oklch:

/* In HSL — shift hue */
color: hsl(from var(--brand) calc(h + 30) s l);

/* In sRGB — halve the red channel */
color: rgb(from var(--brand) calc(r / 2) g b);

/* Make any colour semi-transparent */
color: oklch(from var(--brand) l c h / 0.5);
color: rgb(from var(--brand) r g b / 0.7);

light-dark() — theme-aware colour in one line

light-dark() picks between two values based on the user's colour-scheme preference

I use light-dark() — my background is white in light mode, dark in dark mode. 
<!-- HTML -->
<div class="card">Theme-aware card</div>

/* CSS — one-line theme-aware values */
:root {
  color-scheme: light dark;  /* opt into both modes */
}

.card {
  background: light-dark(white, #1a1a2e);
  color: light-dark(#1a1a2e, #f0f0f0);
  border-color: light-dark(#e0e0e0, #333);
}

/* No @media (prefers-color-scheme) needed!
   light-dark() automatically picks based on the active scheme */

light-dark() replaces the pattern of duplicate @media blocks for dark mode. Instead of defining every colour twice in separate media queries, you define it once with both values inline. Requires color-scheme: light dark on the root.

Implementing theme modes — the full picture

How light-dark() knows which value to pick

light-dark() reads from the color-scheme CSS property on the element (or its ancestors). The color-scheme property tells the browser which mode is active:

/* "I support both — use whatever the OS says" */
:root { color-scheme: light dark; }
/* → light-dark() picks based on the user's OS appearance setting */

/* "Force light mode regardless of OS" */
:root { color-scheme: light; }
/* → light-dark() ALWAYS picks the first value */

/* "Force dark mode regardless of OS" */
:root { color-scheme: dark; }
/* → light-dark() ALWAYS picks the second value */

The resolution order:

  1. OS preference — the user's system setting (macOS Appearance, Windows Personalisation). Detected by prefers-color-scheme.
  2. color-scheme property — tells the browser which modes you support. light dark = follow OS. light or dark alone = override OS.
  3. light-dark() — reads the resolved mode from color-scheme and picks the corresponding value.

The toggle mechanism: To switch modes, you change the color-scheme value. The simplest way: add a class to <html> that overrides it. All light-dark() calls on the page respond automatically — no redefinition of colours needed.

/* CSS */
:root { color-scheme: light dark; }        /* default: follow OS */
:root.force-light { color-scheme: light; } /* user chose light */
:root.force-dark  { color-scheme: dark; }  /* user chose dark */

// JS — toggle is just a class swap
document.documentElement.classList.remove('force-light', 'force-dark');
document.documentElement.classList.add('force-dark');
// → Every light-dark() on the page now picks its second value

There are several approaches to building on this. Here's how they compare:

Approach 1: System preference only (CSS-only, no toggle)

/* Respects the user's OS setting — no manual toggle */
:root {
  color-scheme: light dark;  /* tells the browser both schemes are supported */
}

.card {
  background: light-dark(white, #1a1a2e);
  color: light-dark(#1a1a2e, #f0f0f0);
  border-color: light-dark(#e0e0e0, #333);
}

/* What color-scheme: light dark does:
   - Tells the browser to use the user's OS preference
   - Enables light-dark() to pick the right value
   - Also affects form controls, scrollbars, etc. (they theme automatically)
*/

Approach 2: System preference + user override toggle

Most apps want both: respect the OS default, but let the user override it. The cleanest pattern uses custom properties + a class on the root:

Click to cycle modes — note how the card responds

This card adapts to the theme mode. 
<!-- HTML -->
<html class="light">  <!-- or "dark", or no class for system -->
  <button onclick="cycleTheme()">Toggle</button>
  <div class="card">Content</div>
</html>

/* CSS — define tokens per theme */
:root {
  color-scheme: light dark;

  /* Default: system preference via light-dark() */
  --color-bg: light-dark(white, #1a1a2e);
  --color-text: light-dark(#1a1a2e, #f0f0f0);
  --color-border: light-dark(#e0e0e0, #333);
  --color-surface: light-dark(#f8f8f8, #2a2a3e);
}

/* Override: force light */
:root.light {
  color-scheme: light;
  --color-bg: white;
  --color-text: #1a1a2e;
  --color-border: #e0e0e0;
  --color-surface: #f8f8f8;
}

/* Override: force dark */
:root.dark {
  color-scheme: dark;
  --color-bg: #1a1a2e;
  --color-text: #f0f0f0;
  --color-border: #333;
  --color-surface: #2a2a3e;
}

/* Components just use the tokens — no theme logic here */
.card {
  background: var(--color-bg);
  color: var(--color-text);
  border-color: var(--color-border);
}

// JavaScript — cycle between system/light/dark
const modes = ['system', 'light', 'dark'];
let currentMode = 0;

function cycleTheme() {
  currentMode = (currentMode + 1) % modes.length;
  const mode = modes[currentMode];
  document.documentElement.classList.remove('light', 'dark');
  if (mode !== 'system') {
    document.documentElement.classList.add(mode);
  }
  // Persist choice
  localStorage.setItem('theme', mode);
}

Approach 3: Using container style queries for scoped themes

From Lesson 10 — you can use style queries to scope themes to a specific section without affecting the whole page:

/* Set a theme custom property on a container */
.sidebar {
  --theme: dark;
}

/* Children respond to the container's --theme */
@container style(--theme: dark) {
  .card {
    background: #1a1a2e;
    color: #f0f0f0;
  }
}

/* The sidebar is dark, the rest of the page stays light.
   No duplication of component styles — just one @container rule. */

The recommended architecture

/* 1. Define semantic colour tokens on :root */
:root {
  color-scheme: light dark;

  --color-bg-primary: light-dark(white, #0f0f1a);
  --color-bg-surface: light-dark(#f8f8fa, #1a1a2e);
  --color-text-primary: light-dark(#1a1a2e, #f0f0f0);
  --color-text-secondary: light-dark(#666, #aaa);
  --color-border: light-dark(#e0e0e0, #333);
  --color-accent: oklch(65% 0.2 250);  /* same in both modes */
  --color-accent-subtle: light-dark(
    oklch(from var(--color-accent) 95% 0.03 h),
    oklch(from var(--color-accent) 20% 0.05 h)
  );
}

/* 2. Override with .light / .dark class for user toggle */
:root.light { color-scheme: light; /* override tokens here if needed */ }
:root.dark  { color-scheme: dark;  /* override tokens here if needed */ }

/* 3. Components ONLY reference tokens — never raw colours */
.card {
  background: var(--color-bg-surface);
  color: var(--color-text-primary);
  border: 1px solid var(--color-border);
}

.button {
  background: var(--color-accent);
  color: white;
}

/* Result: change the mode → everything updates.
   Components have zero awareness of themes. */

The principle: Separate the three concerns: (1) token definition (what colours exist), (2) theme mode (which variant is active), (3) component usage (referencing tokens). Components should never contain theme logic — they just use tokens. The root handles switching.
