Lesson 12

CSS Nesting — Native Nesting Without Preprocessors

If you've used Sass or Less, you know nesting: writing child selectors inside parent rules to keep related styles together. CSS now supports this natively — no build step needed.

The mental model

Nesting is purely a convenience for authoring. It doesn't change what the browser does — it just lets you write selectors in a hierarchical structure that mirrors your DOM. The browser de-sugars nested rules into equivalent flat selectors.

Key principle: Nesting changes how you write CSS, not how it resolves. The specificity and cascade behaviour of nested rules is identical to their flattened equivalents. It's syntactic sugar — powerful sugar, but sugar nonetheless.

Basic nesting with &

The & symbol represents the parent selector. Nested rules combine with the parent:

A card component with nested child and state selectors — hover to see the effect

Card Title
This card uses native CSS nesting for all its internal styles.
<!-- HTML -->
<div class="card">
  <div class="title">Card Title</div>
  <div class="body">Content here</div>
</div>

/* Nested CSS */
.card {
  background: white;
  border: 1px solid gray;
  padding: 1rem;

  & .title {           /* → .card .title */
    font-weight: 600;
    color: blue;
  }

  & .body {            /* → .card .body */
    font-size: 0.85rem;
    color: #555;
  }

  &:hover {            /* → .card:hover */
    border-color: blue;
  }
}

Each nested rule with & expands by replacing & with the parent selector. The comments show the equivalent flat selectors.

When & is required vs optional

The & is required when the nested selector starts with something that could be confused with a property name. In practice:

  • & .child — required (starts with . which is fine, but & is clearer)
  • &:hover — required (pseudo-classes must attach to parent with &)
  • &.modifier — required (compound selector, no space between parent and class)
  • & > .direct — required for child combinator

Actually, as of the updated spec (2024+), bare nesting without & works for most cases:

.card {
  /* These are equivalent in modern browsers: */
  & .title { color: blue; }
  .title { color: blue; }      /* also works — & is implicit */

  /* But & IS required for: */
  &:hover { }       /* pseudo-class on the parent */
  &.featured { }    /* adding a class to the parent (no space) */
  & + .sibling { }  /* combinators relative to parent */
}

Best practice: Always use & explicitly. It makes the intent unambiguous, works in all browsers that support nesting, and makes it obvious that you're nesting (vs accidentally writing a standalone selector inside a rule).

Compound selectors: & with no space

&.class (no space) appends to the parent. & .class (with space) is a descendant:

Quick refresher: space vs no space in selectors

This distinction is fundamental to CSS selectors and directly maps to how & works in nesting:

/* SPACE = descendant combinator: .title INSIDE .card (different elements) */
.card .title { }
<div class="card">
  <div class="title">Matches ✓</div>
</div>

/* NO SPACE = compound selector: ONE element with BOTH classes */
.card.title { }
<div class="card title">Matches ✓</div>
Selector Means Nesting equivalent
.card .title .title inside .card (any depth) & .title
.card.featured same element has both classes &.featured
.card > .title .title is a direct child of .card & > .title
.card, .title .card OR .title (selector list) separate rules

The rule: A space between selectors means "look inside" (parent → descendant). No space means "same element, multiple conditions." This is the single most important punctuation in CSS selectors.

.card.featured — compound selector (same element has both classes)

Featured Card
This card has both .card AND .featured on the same element.
<!-- HTML -->
<div class="card featured">...</div>

.card {
  border: 1px solid gray;

  &.featured {         /* → .card.featured (same element) */
    border-color: coral;
    border-width: 2px;

    & .title {          /* → .card.featured .title (descendant) */
      color: coral;
    }
  }
}

Nesting media queries

You can nest @media rules inside a selector — they inherit the parent context:

Grid switches from 1 column to 2 columns at 500px — media query nested inside the component

Item 1
Item 2
Item 3
Item 4
<!-- HTML -->
<div class="grid">
  <div class="item">1</div>
  <div class="item">2</div>
  <div class="item">3</div>
  <div class="item">4</div>
</div>

.grid {
  display: grid;
  grid-template-columns: 1fr;
  gap: 1rem;

  @media (min-width: 500px) {
    grid-template-columns: 1fr 1fr;  /* → @media (...) { .grid { ... } } */
  }
}

Why this matters: Without nesting, the media query and the selector are physically separated — you define .grid in one place and its responsive variant in another @media block elsewhere. With nesting, the responsive behaviour lives right next to the base styles. Co-location = easier to reason about.

Nesting other at-rules

Beyond @media, you can nest @container, @supports, and @layer inside selectors:

@supports is a feature query — it asks the browser "do you understand this property+value?" and only applies the styles if yes. Think of it like a media query, but for CSS feature support instead of viewport size:

.card {
  padding: 1rem;

  @container (min-width: 400px) {
    padding: 2rem;  /* → @container (...) { .card { padding: 2rem; } } */
  }

  @supports (backdrop-filter: blur(10px)) {
    /* Only applies if the browser supports backdrop-filter */
    backdrop-filter: blur(10px);
  }

  @supports not (container-type: inline-size) {
    /* Fallback for browsers without container queries */
    max-width: 600px;
  }
}

@supports is less necessary than it used to be — most modern features degrade gracefully (unknown properties are just ignored). But it's useful when you need a completely different approach as a fallback, or when a set of styles only makes sense together.

Specificity of nested selectors

Nested selectors have the same specificity as their flat equivalent. There's no specificity penalty or bonus for nesting. (For a full primer on how specificity is calculated, see Lesson 11.)

/* These have IDENTICAL specificity: */

/* Flat */
.card .title { color: blue; }

/* Nested */
.card {
  & .title { color: blue; }
}

/* Both = (0, 2, 0) — two class selectors */

Deep nesting doesn't magically increase specificity beyond what the equivalent flat selector would be. But be careful: deeply nested selectors ARE more specific because they have more parts — just like writing the flat equivalent would be.

/* This is high specificity whether nested or flat: */
.page {
  & .main {
    & .content {
      & .card {
        & .title { color: blue; }
      }
    }
  }
}
/* = .page .main .content .card .title → (0, 5, 0) — same as writing it flat */

The trap: Nesting makes it easy to accidentally create deeply nested selectors (matching your DOM nesting depth) that are over-specific. Just because you CAN nest 5 levels deep doesn't mean you should. Keep nesting shallow — 2-3 levels max is a good rule of thumb.

CSS nesting vs Sass nesting — differences

Interpolation (#{$var}) Yes No (can't build selectors dynamically)
@extend Yes No

Browser support

CSS nesting: Chrome 120+, Safari 17.2+, Firefox 117+. The relaxed syntax (implicit &) shipped slightly later in some browsers — using explicit & ensures widest compatibility.
