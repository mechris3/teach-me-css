Lesson 17

Anchor Positioning — Position Relative to Any Element

Traditional CSS positioning gives you three reference frames: the normal flow, a positioned parent (position: relative), or the viewport (position: fixed). But what if you want to position a tooltip relative to a button that's nowhere near it in the DOM? Previously: JavaScript. Now: CSS Anchor Positioning.

The mental model

Two roles:

  • Anchor — the element you're positioning relative TO (the reference point). It gets anchor-name.
  • Positioned element — the element being placed (tooltip, popover, dropdown). It uses position-anchor and anchor() functions to reference the anchor's edges.

The breakthrough: The positioned element doesn't need to be inside or adjacent to the anchor in the DOM. It can be anywhere in the document — it references the anchor by name. This decouples visual position from DOM structure.

Basic anchor positioning — tooltip above a button

The tooltip is positioned relative to the button using anchor() — no JS, no position: relative parent needed

I'm the anchor
Tooltip above the button
<!-- HTML — the tooltip can be anywhere in the DOM -->
<div class="button">I'm the anchor</div>
<div class="tooltip">Tooltip above the button</div>

/* CSS */

/* Step 1: Name the anchor */
.button {
  anchor-name: --my-button;
}

/* Step 2: Position relative to it */
.tooltip {
  position: absolute;
  position-anchor: --my-button;     /* which anchor to reference */
  bottom: anchor(top);              /* my bottom edge = anchor's top edge */
  left: anchor(center);             /* my left edge = anchor's horizontal center */
  translate: -50% 0;                /* shift left by half my width (center me) */
  margin-bottom: 8px;               /* gap between tooltip and anchor */
}

How anchor() works

The anchor() function returns a position value from the anchor element's edges:

/* anchor() takes a side of the anchor element: */
anchor(top)      /* the anchor's top edge position */
anchor(bottom)   /* the anchor's bottom edge position */
anchor(left)     /* the anchor's left edge position */
anchor(right)    /* the anchor's right edge position */
anchor(center)   /* the anchor's center (horizontal or vertical depending on property) */

/* Used in inset properties: */
top: anchor(bottom);       /* "my top = anchor's bottom" → below the anchor */
bottom: anchor(top);       /* "my bottom = anchor's top" → above the anchor */
left: anchor(right);       /* "my left = anchor's right" → to the right of anchor */
right: anchor(left);       /* "my right = anchor's left" → to the left of anchor */

Reading anchor() declarations: The property you're setting is YOUR edge. The value inside anchor() is THEIR edge. So top: anchor(bottom) means "place my top edge at their bottom edge" — i.e. position me below them.

/* Spanning values: */
/* position-area: top span-all;    → above, full width */
/* position-area: center span-all; → same row as anchor, full width */
/* position-area: span-all left;   → left column, full height */

The anchor follows the target — live demo

Drag the blue button — the tooltip follows it automatically (pure CSS positioning)

Drag me
I follow via CSS!
<!-- HTML -->
<div class="button" id="draggable">Drag me</div>
<div class="tooltip">I follow via CSS!</div>

/* CSS — the tooltip is anchored to the button */
.button {
  anchor-name: --drag-btn;
  position: absolute;  /* moved by JS drag */
}

.tooltip {
  position: absolute;
  position-anchor: --drag-btn;
  bottom: anchor(top);
  left: anchor(center);
  translate: -50% 0;
  margin-bottom: 8px;
}

// JavaScript — only handles the drag. Tooltip follows automatically via CSS.
const btn = document.getElementById('draggable');
let isDragging = false, offsetX, offsetY;

btn.addEventListener('pointerdown', (e) => {
  isDragging = true;
  offsetX = e.offsetX;
  offsetY = e.offsetY;
  btn.setPointerCapture(e.pointerId);
});

document.addEventListener('pointermove', (e) => {
  if (!isDragging) return;
  const area = btn.parentElement.getBoundingClientRect();
  btn.style.left = (e.clientX - area.left - offsetX) + 'px';
  btn.style.top = (e.clientY - area.top - offsetY) + 'px';
});

document.addEventListener('pointerup', () => { isDragging = false; });

The key insight here: The JavaScript only moves the button. The tooltip has zero JS — it follows the button purely through CSS anchor positioning. If the anchor moves (via drag, scroll, resize, animation), the positioned element tracks it automatically.

position-try — fallback positions

What if the tooltip would overflow the viewport? position-try-fallbacks lets you define fallback positions the browser can try automatically:

The button is near the top — the tooltip prefers to be above, but flips below because there's no room above

Near the top edge
I flip below when no room above
<!-- HTML -->
<div class="button">Near the top edge</div>
<div class="tooltip">I flip below when no room above</div>

/* CSS */
.button { anchor-name: --edge-btn; }

.tooltip {
  position: absolute;
  position-anchor: --edge-btn;
  bottom: anchor(top);          /* preferred: above the button */
  left: anchor(center);
  translate: -50% 0;
  margin-bottom: 8px;

  /* If above doesn't fit (overflows container), try below */
  position-try-fallbacks: flip-block;
}

/* position-try-fallbacks values: */
/* flip-block        → flip vertically (above ↔ below) */
/* flip-inline       → flip horizontally (left ↔ right) */
/* flip-block flip-inline → try both flips */

/* Or define a CUSTOM fallback position: */
@position-try --below {
  bottom: auto;               /* clear the "above" positioning */
  top: anchor(bottom);        /* position below instead */
  margin-bottom: 0;
  margin-top: 8px;
}

.tooltip {
  position-try-fallbacks: --below;  /* use custom fallback */
}

Why this matters: Previously, tooltip/popover libraries used JS to detect overflow and flip position. Anchor positioning with position-try-fallbacks does this natively in CSS — the browser checks if the element overflows its containing block and automatically tries fallback positions.

anchor-name rules

  • Names must start with -- (just like custom properties)
  • Names must be unique within the anchor scope — two elements can't share the same anchor-name
  • The anchor must be visible (not display: none) for positioning to work
  • The positioned element must be position: absolute or position: fixed

z-index, stacking contexts, and the Popover API

The z-index problem

Anchor-positioned elements are position: absolute, which means they participate in the stacking context of their containing block. This creates a classic gotcha:

/* The problem scenario: */
.sidebar {
  position: relative;
  z-index: 1;  /* creates a stacking context */
}

.tooltip {
  position: absolute;
  position-anchor: --some-button;
  z-index: 9999;  /* doesn't matter! trapped in sidebar's stacking context */
}

/* The tooltip can't appear above content outside .sidebar,
   because z-index only competes WITHIN the same stacking context.
   The sidebar's z-index: 1 caps everything inside it. */

The rule: z-index only competes with elements in the same stacking context. A stacking context is created by any element with: position + z-index, opacity < 1, transform, filter, will-change, or being a flex/grid child with z-index. Once trapped in a lower stacking context, no z-index value can break out.

The solution: the Popover API + top layer

The Popover API is an HTML feature that solves the z-index problem. An element with the popover attribute renders in the browser's top layer — a special layer that sits above ALL stacking contexts. Nothing can z-index over it.

Popover API deep-dive

The Popover API is primarily HTML-driven — you add attributes to elements and the browser handles show/hide, focus, and layering:

Click the button — the popover opens anchored below it (light dismiss: click outside or Escape to close)

I'm a popover!

No JS needed. Click outside or press Escape to dismiss. I'm in the top layer AND anchored to the button.

<!-- HTML — minimal popover + anchor positioning combined -->
<button popovertarget="my-popup">Open</button>
<div id="my-popup" popover>
  I'm a popover! Click outside to dismiss.
</div>

/* CSS — anchor it to the button */
button {
  anchor-name: --pop-btn;
}

[popover] {
  position: absolute;
  position-anchor: --pop-btn;
  top: anchor(bottom);
  left: anchor(left);
  margin-top: 4px;
}

HTML attributes:

Attribute Goes on What it does
popover The popover element Marks it as a popover (hidden by default, renders in top layer when open)
popover="auto" The popover element Default — light dismiss (closes on outside click/Escape), only one open at a time
popover="manual" The popover element No light dismiss — must be closed explicitly (multiple can be open)
popovertarget="id" The trigger button Clicking this button toggles the popover with the matching ID
popovertargetaction The trigger button "toggle" (default), "show", or "hide"

What you get for free (no JS):

  • Top layer — above all stacking contexts, no z-index needed
  • Light dismiss — clicking outside or pressing Escape closes it (for popover="auto")
  • One-at-a-time — opening a new auto popover closes the previous one
  • Focus trap — keyboard focus stays manageable
  • Accessible — proper ARIA semantics built in
  • ::backdrop — styleable backdrop behind the popover (optional)
  • :popover-open — CSS pseudo-class for styling the open state

JS API (when you need programmatic control):

const popup = document.getElementById('my-popup');

popup.showPopover();     // open it
popup.hidePopover();     // close it
popup.togglePopover();   // toggle

// Events
popup.addEventListener('toggle', (e) => {
  console.log(e.newState);  // 'open' or 'closed'
});

CSS for styling popovers:

/* Style when open */
[popover]:popover-open {
  opacity: 1;
  transform: scale(1);
}

/* Style when closed (for entry/exit animation) */
[popover] {
  opacity: 0;
  transform: scale(0.95);
  transition: opacity 0.2s, transform 0.2s, display 0.2s allow-discrete;
}

/* Style the backdrop (only visible when popover is open) */
[popover]::backdrop {
  background: rgba(0, 0, 0, 0.3);
  backdrop-filter: blur(2px);
}

Browser support: Chrome 114+, Safari 17+, Firefox 125+. Fully production-ready.

Combining popover + anchor positioning

Click the button — the popover opens in the top layer, positioned via anchor CSS

Popover content

I'm in the top layer — z-index battles are irrelevant. Click outside to dismiss.

<!-- HTML -->
<button popovertarget="my-popup">Open popover ▾</button>
<div id="my-popup" popover>
  Popover content — click outside to dismiss
</div>

/* CSS — anchor the popover to the button */
button {
  anchor-name: --pop-trigger;
}

[popover] {
  /* Popover is already in the top layer — no z-index needed */
  position: absolute;
  position-anchor: --pop-trigger;
  top: anchor(bottom);
  left: anchor(left);
  margin-top: 4px;
  position-try-fallbacks: flip-block;  /* flip above if no room */
}

/* Style the open/closed states */
[popover]:popover-open {
  opacity: 1;
  transform: translateY(0);
}

/* Optional: animate the entry */
[popover] {
  opacity: 0;
  transform: translateY(-4px);
  transition: opacity 0.2s, transform 0.2s, display 0.2s allow-discrete;
}

When to use popover vs plain absolute

Use popover + anchor when... Use plain position: absolute when...
Needs to appear above everything (tooltips, dropdowns, modals) Stays within its own visual area (badge on a card)
Needs light dismiss (close on outside click) Always visible, doesn't need show/hide
Could be trapped behind stacking contexts No z-index conflicts in your layout

The practical rule: If your anchor-positioned element is a popover, dropdown, or tooltip that needs to reliably appear above all page content — use the popover attribute. It puts the element in the top layer for free, and you get light-dismiss and accessibility baked in. Anchor positioning handles the "where", popover handles the "above everything."

Dynamic anchors — lists and repeated elements

Anchor names must be unique, so what happens when you have a dynamic list (e.g. table rows from an API) where each row has a button that needs an anchored popover?

The pattern: one shared popover, many anchors

The most practical approach: give each button a unique anchor name via inline style binding, and re-point a single shared popover at whichever anchor was clicked.

<!-- Angular template -->

<!-- Each row's button gets a unique anchor name -->
<tr *ngFor="let row of data">
  <td>{{ row.name }}</td>
  <td>
    <button
      [style.anchor-name]="'--btn-' + row.id"
      (click)="openMenu(row.id)">
      Actions ▾
    </button>
  </td>
</tr>

<!-- ONE shared popover — repositioned dynamically -->
<div popover id="shared-menu"
     [style.position-anchor]="'--btn-' + activeRowId">
  <a href="#">Edit</a>
  <a href="#">Delete</a>
  <a href="#">Duplicate</a>
</div>

/* CSS — positioning is static, only the anchor reference changes */
[popover] {
  position: absolute;
  top: anchor(bottom);
  left: anchor(left);
  margin-top: 4px;
  position-try-fallbacks: flip-block;
}

// TypeScript
activeRowId = '';

openMenu(rowId: string) {
  this.activeRowId = rowId;
  document.getElementById('shared-menu')?.showPopover();
}

Why this works: The CSS positioning rules (top: anchor(bottom), etc.) are written once and never change. Only the position-anchor value changes — pointing at a different anchor name on each click. One popover element, N anchor buttons, zero positioning JS.

Alternative: the HTML anchor attribute (no CSS name needed)

The spec also defines an anchor HTML attribute that creates an implicit anchor relationship by element ID — no anchor-name CSS needed:

<!-- HTML — implicit anchoring via the anchor attribute -->
<button id="btn-42" popovertarget="menu-42">Actions</button>
<div popover id="menu-42" anchor="btn-42">
  Menu for row 42
</div>

/* CSS — no anchor-name needed; the anchor attribute does the linking */
[popover] {
  position: absolute;
  top: anchor(bottom);
  left: anchor(left);
  margin-top: 4px;
}

The downside: this requires one popover element per row (heavier DOM). The shared-popover pattern above is lighter for lists.

When inline anchor-name is acceptable

Setting anchor-name via inline style (framework binding) is an acceptable use of inline styles — the same pattern as view-transition-name per list item. The value is data-driven and unique per element, which means it can't be expressed in a static stylesheet. CSS owns the positioning; the framework just supplies the unique identifier.

Browser support

Anchor positioning: Chrome 125+, Edge 125+. Not yet in Firefox or Safari (as of mid-2025). Progressive enhancement — without support, elements fall back to their normal position or can be positioned with a JS polyfill.
