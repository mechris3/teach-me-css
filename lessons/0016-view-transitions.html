View Transitions — Animating Between States

View transitions let you animate between two DOM states — whether that's toggling a UI panel, navigating between pages, or reordering a list. The browser snapshots the "before" state, you make the DOM change, and then it animates from the snapshot to the new state. The result is smooth, app-like transitions with very little code.

The mental model

A view transition works in three phases:

  1. Snapshot — the browser captures an image of the current state (the "old" view)
  2. DOM update — you make your changes (swap content, toggle classes, navigate)
  3. Animate — the browser crossfades from the snapshot to the new state, with optional custom animations

Key concept: The browser does the heavy lifting. You don't manually animate properties between states. You say "I'm about to change something" → change it → the browser figures out how to animate the difference. It's declarative animation of state changes.

Two types of view transitions

  • Same-document (SPA) — triggered with document.startViewTransition() in JavaScript. For single-page apps where you're updating the DOM without a full navigation.
  • Cross-document (MPA) — triggered automatically on navigation between pages that both opt in with @view-transition { navigation: auto; }. For multi-page apps and traditional websites.

Same-document transitions (SPA)

The simplest usage — wrap a DOM change in document.startViewTransition():

Click the button — the card crossfades between two states via view transition

State A — blue background
<!-- HTML -->
<button onclick="toggleCard()">Toggle</button>
<div class="container state-a" id="myContainer">
  <div class="card" id="myCard">State A</div>
</div>

/* CSS — give the element a view-transition-name */
.card {
  view-transition-name: demo-card;
}

.container.state-a { background: lightblue; }
.container.state-b { background: lightsalmon; }

/* Customise the transition animation (optional) */
::view-transition-old(demo-card) {
  animation: fade-out 0.3s ease;
}
::view-transition-new(demo-card) {
  animation: fade-in 0.3s ease;
}

@keyframes fade-out { to { opacity: 0; } }
@keyframes fade-in { from { opacity: 0; } }

// JavaScript — wrap the DOM change
let isStateA = true;

function toggleCard() {
  const container = document.getElementById('myContainer');
  const card = document.getElementById('myCard');

  document.startViewTransition(() => {
    isStateA = !isStateA;
    container.classList.toggle('state-a', isStateA);
    container.classList.toggle('state-b', !isStateA);
    card.textContent = isStateA ? 'State A — blue' : 'State B — orange';
  });
}

How it works — the full timeline

Here's exactly what happens, from the moment you call startViewTransition() to the moment the animation finishes. Understanding this timeline is key to knowing when you can interact with the page, when the DOM is frozen, and what the user actually sees:

TIME ─────────────────────────────────────────────────────────────────────────►

│ YOU CALL                                                                     │
│ document.startViewTransition(callback)                                       │
│                                                                              │
├─── PHASE 1: CAPTURE OLD STATE ──────────────────────────────────────────────┤
│                                                                              │
│  1. The browser FREEZES rendering (the page visually pauses)                │
│  2. It finds all elements with a view-transition-name                        │
│  3. It captures a bitmap snapshot of each named element                      │
│     (position, size, appearance — like a screenshot)                         │
│  4. It also captures the entire page as the "root" snapshot                  │
│                                                                              │
│  → transition.ready has NOT resolved yet                                     │
│  → The user sees: the frozen page (nothing has changed visually)             │
│                                                                              │
├─── PHASE 2: UPDATE THE DOM ─────────────────────────────────────────────────┤
│                                                                              │
│  5. Your callback() runs — you make ALL your DOM changes here               │
│     (toggle classes, swap content, add/remove elements, etc.)                │
│  6. The browser waits for your callback to complete                          │
│     (if it returns a Promise, it waits for that too)                         │
│  7. The browser renders the new DOM state (off-screen — user doesn't see it)│
│  8. It captures new snapshots of all named elements in their new positions   │
│                                                                              │
│  → transition.updateCallbackDone resolves here                               │
│  → The user STILL sees: the old frozen snapshot (hasn't changed yet)         │
│                                                                              │
├─── PHASE 3: CREATE PSEUDO-ELEMENTS ─────────────────────────────────────────┤
│                                                                              │
│  9. The browser creates the pseudo-element tree:                             │
│     ::view-transition                                                        │
│       └── ::view-transition-group(name)                                      │
│             └── ::view-transition-image-pair(name)                            │
│                   ├── ::view-transition-old(name)  ← old snapshot bitmap     │
│                   └── ::view-transition-new(name)  ← new snapshot bitmap     │
│                                                                              │
│  10. These pseudo-elements are placed in a top layer above the page          │
│      (like a dialog — nothing on the page is interactive)                    │
│                                                                              │
│  → transition.ready resolves HERE                                            │
│  → You can now add classes to control animation direction                    │
│  → The user sees: the old snapshot (about to start animating)                │
│                                                                              │
├─── PHASE 4: ANIMATE ────────────────────────────────────────────────────────┤
│                                                                              │
│  11. Default animation runs:                                                 │
│      - ::view-transition-old fades out (opacity 1 → 0)                       │
│      - ::view-transition-new fades in (opacity 0 → 1)                        │
│      - If same name exists in old+new: GROUP morphs position/size            │
│  12. Your custom @keyframes override the default if specified                │
│  13. Duration is 0.25s by default (customisable via animation-duration)      │
│                                                                              │
│  → The user sees: the animation playing (pseudo-element images, not DOM)     │
│  → The page is NOT interactive during this phase                             │
│                                                                              │
├─── PHASE 5: CLEANUP ────────────────────────────────────────────────────────┤
│                                                                              │
│  14. Animation completes                                                     │
│  15. All pseudo-elements are removed                                         │
│  16. The new DOM is revealed underneath (it was there all along)             │
│  17. The page becomes interactive again                                      │
│                                                                              │
│  → transition.finished resolves HERE                                         │
│  → The user sees: the new DOM state, live and interactive                    │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘

Key details to note

  • The page is NOT interactive during the animation. The user sees images (snapshots), not the real DOM. Keep transitions short (0.2–0.5s) to avoid feeling "frozen."
  • The DOM change is instant. Your callback runs synchronously — the new DOM exists immediately. The animation is purely visual over the top.
  • If your callback is async (returns a Promise), the browser waits. Useful if you need to fetch data before showing the new state.
  • Rendering is frozen from step 1 to step 10. If your callback is slow, the freeze is perceptible. Keep the DOM update fast.
  • "Same name" elements get a morph, not a crossfade. The ::view-transition-group for that name smoothly interpolates width, height, and transform between the old and new positions. This is what makes shared elements feel like they "move."
  • Elements without a name still participate. The entire page is captured as the root transition. Named elements are extracted and animated independently on top of the root crossfade.

The promises — when to use each

const transition = document.startViewTransition(async () => {
  await fetchNewData();       // can be async
  updateDOM();
});

// Resolves when your DOM update is done (phase 2 complete)
transition.updateCallbackDone.then(() => {
  console.log('DOM updated');
});

// Resolves when pseudo-elements are created (animation about to start)
transition.ready.then(() => {
  // Good place to add directional classes (slide-left vs slide-right)
  document.documentElement.classList.add('navigating-forward');
});

// Resolves when animation finishes and page is interactive again
transition.finished.then(() => {
  document.documentElement.classList.remove('navigating-forward');
  console.log('Transition complete — page is live');
});

The magic: During the animation, the user sees an image (the snapshot), not the actual DOM. This means the DOM change can be instant and jarring — the view transition smooths it visually. Even a complete content swap looks elegant.

view-transition-name — identifying elements across states

This is the core CSS property. It tells the browser: "this element in the old state corresponds to this element in the new state — animate between them."

.card {
  view-transition-name: hero-card;  /* must be unique on the page */
}

.avatar {
  view-transition-name: user-avatar;
}

/* RULE: each name must be unique per page.
   Two elements with the same name = error (browser can't know which is which) */

When the browser finds an element with the same view-transition-name in both the old and new states, it animates between their positions/sizes. This is how you get "shared element" transitions — like a thumbnail morphing into a full-size image.

Live demo: shared element transition

Click a thumbnail — it expands into a full-width panel (shared element transition)

<!-- HTML -->
<div class="grid">
  <div class="thumb" style="view-transition-name: thumb-1"></div>
  <div class="thumb" style="view-transition-name: thumb-2"></div>
  <div class="thumb" style="view-transition-name: thumb-3"></div>
</div>
<div class="expanded" style="view-transition-name: thumb-1"></div>

/* The key: both the thumbnail and the expanded view share
   the same view-transition-name. The browser morphs between them. */

.thumb:nth-child(1) { view-transition-name: thumb-1; }
.expanded         { view-transition-name: thumb-1; }  /* same name! */

// JS
function expandThumb(index) {
  document.startViewTransition(() => {
    // Hide grid, show expanded — browser animates the morph
    grid.style.display = 'none';
    expanded.style.display = 'flex';
    expanded.style.viewTransitionName = `thumb-${index + 1}`;
  });
}

Shared elements must have the same view-transition-name in both states. The browser matches old→new by name. When it finds a match, instead of crossfading, it morphs: smoothly interpolating position, size, and opacity between the old and new locations. This gives you native "hero" animations without any animation code.

The pseudo-element tree

During a view transition, the browser creates a tree of pseudo-elements you can style. These are pseudo-elements (double colon ::) not pseudo-classes (single colon :) — the distinction:

  • :hover, :focus, :has() — pseudo-classes (select a real element in a particular state)
  • ::before, ::after, ::view-transition-old() — pseudo-elements (target something the browser generates that doesn't exist in your HTML)

The tree structure

::view-transition
├── ::view-transition-group(name)
│   ├── ::view-transition-image-pair(name)
│   │   ├── ::view-transition-old(name)   ← the "before" snapshot
│   │   └── ::view-transition-new(name)   ← the "after" snapshot

What's in the parentheses?

The value in parentheses (e.g. demo-card) is a selector filter — it references the view-transition-name you assigned to an element. It says "target the pseudo-element for this specific named transition."

/* You assign a name to an element: */
.card { view-transition-name: demo-card; }

/* Then you target that name's pseudo-elements: */
::view-transition-old(demo-card) { }   ← "old snapshot OF demo-card"
::view-transition-new(demo-card) { }   ← "new snapshot OF demo-card"

/* Special values: */
::view-transition-old(root)  { }       ← the whole-page transition
::view-transition-group(*)   { }       ← ALL named transitions at once

The pseudo-element names themselves (::view-transition-old, ::view-transition-new, ::view-transition-group) are fixed by the spec. The parenthesised part is your chosen name — linking the pseudo-element to a specific element's transition.

You can target these to customise the animation for specific named elements:

Custom animation: the old state slides out left, the new state slides in from the right

Panel A content 
<!-- HTML -->
<button onclick="toggleSlide()">Toggle with slide</button>
<div class="slide-panel" id="slideContainer">Panel A content</div>

/* CSS — name the element + define custom animations */
.slide-panel {
  view-transition-name: slide-demo;
  background: #f0f7ff;
  padding: 1rem;
  border-radius: 6px;
  min-height: 60px;
  display: flex;
  align-items: center;
  justify-content: center;

  animation: slide-out-left 0.3s ease forwards;
}

::view-transition-new(slide-demo) {
  animation: slide-in-right 0.3s ease forwards;
}

@keyframes slide-out-left {
  to { transform: translateX(-100%); opacity: 0; }
}

@keyframes slide-in-right {
  from { transform: translateX(100%); opacity: 0; }
}

// JavaScript — trigger the transition
let isSlideA = true;

function toggleSlide() {
  const container = document.getElementById('slideContainer');

  document.startViewTransition(() => {
    isSlideA = !isSlideA;
    container.textContent = isSlideA ? 'Panel A content' : 'Panel B content';
    container.style.background = isSlideA ? '#f0f7ff' : '#fff3e0';
  });
}

Targeting all transitions (the root)

The root name refers to the whole-page transition — everything not given a specific view-transition-name. You can customise it like any other named transition:

<!-- HTML — any page content -->
<button onclick="swapContent()">Change page content</button>
<main id="content">
  <p>Original content here.</p>
</main>

/* CSS — customise the root (whole-page) transition */

/* Make the old page fade out fast */
::view-transition-old(root) {
  animation: fade-out 0.2s ease forwards;
}

/* Make the new page fade in slower */
::view-transition-new(root) {
  animation: fade-in 0.4s ease forwards;
}

@keyframes fade-out { to { opacity: 0; } }
@keyframes fade-in { from { opacity: 0; } }

/* Or: disable the default crossfade entirely (no animation) */
::view-transition-group(root) {
  animation: none;
}

// JavaScript — trigger the transition
function swapContent() {
  document.startViewTransition(() => {
    document.getElementById('content').innerHTML = '<p>New content!</p>';
  });
}

Cross-document transitions (MPA)

For traditional multi-page navigations (clicking links that load a new HTML page), you can opt in with a CSS at-rule — no JavaScript needed:

This is a CSS-only opt-in — both the source and destination pages need this rule

Add @view-transition { navigation: auto; } to both pages → navigations get a smooth crossfade automatically. 
/* Add to BOTH pages (source and destination) */
@view-transition {
  navigation: auto;
}

/* Elements with matching view-transition-name on both pages
   will morph between positions during navigation */
.page-header {
  view-transition-name: header;  /* same name on both pages → shared element */
}

.hero-image {
  view-transition-name: hero;    /* thumbnail on page A, full image on page B */
}

Cross-document transitions are huge for Angular. In Angular, route changes are DOM updates within the same document — so Angular uses same-document view transitions. The Angular Router wraps route activation inside document.startViewTransition() for you, so you don't call it manually.

Angular integration — withViewTransitions() in depth

Angular has built-in view transition support since v17. The Router handles the startViewTransition() call — you just opt in and write CSS.

Enabling view transitions

View transitions are configured at the router level, not per-component. You enable it once in your app config — then all route navigations get transitions automatically.

// app.config.ts — standalone component bootstrap (Angular 17+, recommended)
import { ApplicationConfig } from '@angular/core';
import { provideRouter, withViewTransitions } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes, withViewTransitions())
  ]
};

// main.ts — bootstraps the standalone root component
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';

bootstrapApplication(AppComponent, appConfig);

That's the full setup for a standalone-component app. No modules needed. The withViewTransitions() feature is passed to provideRouter alongside your routes.

If you're in a legacy NgModule-based app:

// app-routing.module.ts — NgModule approach (legacy)
@NgModule({
  imports: [
    RouterModule.forRoot(routes, { enableViewTransitions: true })
  ],
  exports: [RouterModule]
})
export class AppRoutingModule {}

Both do the same thing — the standalone version is just the modern Angular way of configuring providers without a module wrapper.

How the Router uses it — the lifecycle

  1. User triggers navigation (routerLink click, router.navigate(), etc.)
  2. Router runs guards, resolvers, and lazy-loads the component
  3. Router calls document.startViewTransition()
  4. Inside the callback: the old route's component is destroyed, the new route's component is created and rendered
  5. Browser captures new state, animates the transition
  6. Transition completes — new page is interactive

Progressive enhancement: If the browser doesn't support view transitions (Firefox as of mid-2025), the navigation still works normally — just without animation. No fallback code needed.

Adding shared element transitions between routes

<!-- products-list.component.html -->
<div *ngFor="let product of products" class="product-card"
     [style.view-transition-name]="'product-' + product.id">
  <img [src]="product.image">
  <h3>{{ product.name }}</h3>
</div>

<!-- product-detail.component.html -->
<div class="product-detail"
     [style.view-transition-name]="'product-' + product.id">
  <img [src]="product.image">
  <h1>{{ product.name }}</h1>
  <p>{{ product.description }}</p>
</div>

/* global-styles.css — MUST be in global styles, not component styles */
::view-transition-group(product-*) {
  animation-duration: 0.3s;
  animation-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
}

Critical: animations go in GLOBAL styles. Angular's view encapsulation scopes component styles, which prevents them from targeting ::view-transition-* pseudo-elements. Always define view transition CSS in styles.css (your global stylesheet), not in component styleUrls.

Dynamic view-transition-name with Angular bindings

Use Angular's property binding to set unique names per item:

<!-- Using style binding (recommended) -->
<div [style.view-transition-name]="'card-' + item.id">

<!-- Using host binding in the component -->
@Component({
  host: {
    '[style.view-transition-name]': "'detail-' + productId()"
  }
})
export class ProductDetailComponent {
  productId = input.required<string>();
}

Advanced control with onViewTransitionCreated

The withViewTransitions() config accepts a callback that fires before each transition. You get access to the ViewTransition object and both route snapshots:

// app.config.ts
import { provideRouter, withViewTransitions, Router, isActive } from '@angular/router';
import { inject } from '@angular/core';

provideRouter(routes, withViewTransitions({
  onViewTransitionCreated: ({ transition, from, to }) => {
    const router = inject(Router);

    // Example 1: Skip transition for same-page fragment navigation
    const targetUrl = router.currentNavigation()!.finalUrl!;
    if (isActive(targetUrl, router, { paths: 'exact', queryParams: 'ignored', fragment: 'ignored' })()) {
      transition.skipTransition();
    }

    // Example 2: Add directional class based on navigation
    const fromIndex = from?.data?.['index'] ?? 0;
    const toIndex = to?.data?.['index'] ?? 0;
    const direction = toIndex > fromIndex ? 'forward' : 'back';

    transition.ready.then(() => {
      document.documentElement.classList.add(direction);
    });
    transition.finished.then(() => {
      document.documentElement.classList.remove(direction);
    });
  }
}))

/* global-styles.css — directional slide animations */
.forward::view-transition-old(root) {
  animation: slide-out-left 0.3s ease forwards;
}
.forward::view-transition-new(root) {
  animation: slide-in-right 0.3s ease forwards;
}
.back::view-transition-old(root) {
  animation: slide-out-right 0.3s ease forwards;
}
.back::view-transition-new(root) {
  animation: slide-in-left 0.3s ease forwards;
}

@keyframes slide-out-left { to { transform: translateX(-100%); opacity: 0; } }
@keyframes slide-in-right { from { transform: translateX(100%); opacity: 0; } }
@keyframes slide-out-right { to { transform: translateX(100%); opacity: 0; } }
@keyframes slide-in-left { from { transform: translateX(-100%); opacity: 0; } }

What onViewTransitionCreated gives you

Property Type Use for
transition ViewTransition Access .ready, .finished, .skipTransition()
from ActivatedRouteSnapshot Source route data, params, path
to ActivatedRouteSnapshot Destination route data, params, path

The callback runs in an injection context, so you can use inject() to access services.

.item:nth-child(1) { view-transition-name: item-1; }
.item:nth-child(2) { view-transition-name: item-2; }
.item:nth-child(3) { view-transition-name: item-3; }
.item:nth-child(4) { view-transition-name: item-4; }

// JS — shuffle and let view transitions handle the animation
function shuffleList() {
  document.startViewTransition(() => {
    const list = document.querySelector('.list');
    const items = [...list.children];
    items.sort(() => Math.random() - 0.5);
    items.forEach(item => list.appendChild(item));
  });
}

Controlling the default transition

The default view transition is a simple crossfade of the entire page (the root element). You can customise timing, easing, and duration. This is CSS-only — no extra JS needed beyond the startViewTransition() call:

<!-- HTML — any page with a view transition triggered -->
<button onclick="document.startViewTransition(() => { /* ... */ })">Go</button>

/* CSS — override default timing on the whole-page transition */
::view-transition-group(root) {
  animation-duration: 0.5s;  /* slower crossfade */
}

/* Or different speeds for old/new */
::view-transition-old(root) {
  animation-duration: 0.2s;  /* old fades out fast */
}
::view-transition-new(root) {
  animation-duration: 0.4s;  /* new fades in slower */
}

/* For a specific named element */
::view-transition-group(hero) {
  animation-duration: 0.4s;
  animation-timing-function: cubic-bezier(0.4, 0, 0.2, 1);
}

Respecting reduced motion

/* Always include this — disable view transition animations for users
   who prefer reduced motion */
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}

Gotchas and limitations

  • Unique names required. Two elements with the same view-transition-name on a page at the same time = error. If you have a list, each item needs a unique name.
  • Only works in the top-level document. View transitions don't work inside iframes.
  • Snapshot is a flat image. During the animation, you can't interact with the page — it's showing an image, not the live DOM. Keep transitions short.
  • Cross-document requires same origin. You can't animate transitions between different domains.
  • Performance. The browser composites the snapshots on the GPU — so it's fast. But capturing large/complex pages can have a brief delay before the animation starts.

Browser support

  • Same-document (startViewTransition): Chrome 111+, Safari 18+, Firefox 🚧
  • Cross-document (@view-transition): Chrome 126+, Safari 18+, Firefox 🚧
  • Fully progressive — if unsupported, the DOM change still happens, just without animation
