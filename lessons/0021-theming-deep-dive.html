Lesson 21

Theming Deep-Dive — Token Architecture & Multi-Brand Systems

Lesson 18 introduced light-dark(), color-scheme, and the basic toggle mechanism. This lesson goes deeper: how to architect a token system that scales to multiple brands, dark/light modes, and scoped theme regions — all in pure CSS, with patterns that map directly to Angular.

The three-tier token model

Mature design systems converge on three layers of tokens. Each layer adds meaning on top of the one below:

Three tiers: primitive → semantic → component

Primitive

--blue-500: #3a86ff --gray-100: #f4f4f8 --space-4: 1rem

Raw values. No opinion on usage. Never referenced directly in components.

Semantic

--color-primary: var(--blue-500) --color-surface: var(--gray-100) --space-inline: var(--space-4)

Purpose-based. Says what it's for, not what it looks like. This is what components consume.

Component

--btn-bg: var(--color-primary) --btn-radius: 4px --card-padding: var(--space-inline)

Per-component overrides. Optional — only needed when a component diverges from semantic defaults.

/* === TIER 1: Primitives — raw palette, never used directly in components === */
:root {
  --blue-500: #3a86ff;
  --blue-700: #1a5fd4;
  --coral-500: #e76f51;
  --gray-50: #fefefe;
  --gray-100: #f4f4f8;
  --gray-800: #1a1a2e;
  --gray-900: #0d0d1a;
  --space-2: 0.5rem;
  --space-4: 1rem;
  --space-6: 1.5rem;
  --radius-sm: 4px;
  --radius-md: 8px;
}

/* === TIER 2: Semantic — what the token is FOR === */
:root {
  --color-primary: var(--blue-500);
  --color-primary-hover: var(--blue-700);
  --color-surface: var(--gray-50);
  --color-surface-raised: var(--gray-100);
  --color-text: var(--gray-800);
  --color-text-muted: #666;
  --color-border: #e0e0e0;
  --space-inline: var(--space-4);
  --space-block: var(--space-6);
  --radius-default: var(--radius-md);
}

/* === TIER 3: Component — per-component overrides (optional) === */
.btn {
  --btn-bg: var(--color-primary);
  --btn-text: white;
  --btn-radius: var(--radius-sm);
  background: var(--btn-bg);
  color: var(--btn-text);
  border-radius: var(--btn-radius);
  padding: var(--space-2) var(--space-4);
}

Why three tiers? Each layer is a contract boundary:

• Change a primitive and every semantic token referencing it updates (palette refresh).

• Swap the semantic layer and every component updates without touching component code (theming/dark mode).

• Override a component token to customise one component without affecting the global system.

Components only reference semantic or component tokens. They never reach down to primitives. This indirection is what makes theming possible.

Dark mode via semantic token swapping

The key insight from Lesson 18: color-scheme drives light-dark(). But for a full token system, you swap token layers rather than using light-dark() on every property. Which layer you swap depends on what you're theming:

  • Dark mode (same brand, different lightness) → swap the semantic layer. Primitives stay the same (your palette still has both light and dark values). What changes is which primitive each semantic token points to: --color-surface maps to --gray-50 in light, --gray-900 in dark.
  • Multi-brand (different brand identity) → each brand provides its own primitives file with brand-appropriate names (Brand A defines --blue-500, Brand B defines --coral-500), and each brand also maps those to the same semantic tokens (--color-primary). Alternatively, use neutral primitive names like --brand-500 that don't claim a specific colour. Either way, component code only references semantic tokens and never changes per brand.

Why dark mode swaps semantics, not primitives

You might wonder: why not just redefine the primitives for dark mode? Here's what that would look like:

/* WRONG approach: swapping primitives for dark mode */
[data-theme="dark"] {
  --gray-50: #1a1a2e;   /* "gray-50" is now... dark?! */
  --gray-900: #fefefe;  /* "gray-900" is now... white?! */
}

This breaks immediately. Primitive names describe what the value IS — --gray-50 means "a very light grey." If you redefine it to be dark, the name becomes a lie. Anyone reading the code (including future you) sees --gray-50 and assumes "light," but it's actually dark. You've turned your primitives into semantic tokens with misleading names.

The deeper issue: in dark mode, you still need both light and dark values available simultaneously. Dark mode text is light (--gray-50), dark mode surfaces are dark (--gray-900). If you redefine --gray-50 to be dark, you've lost access to the light value you need for text.

The correct approach: keep primitives truthful (they always describe their actual value), and let the semantic layer decide which primitive to use in each context:

/* Primitives — ALWAYS truthful, never redefined per theme */
:root {
  --gray-50: #fefefe;    /* always light */
  --gray-900: #0d0d1a;   /* always dark */
}

/* Semantic layer — re-maps which primitives are used */
[data-theme="light"] {
  --color-surface: var(--gray-50);    /* light surface */
  --color-text: var(--gray-900);      /* dark text on light */
}
[data-theme="dark"] {
  --color-surface: var(--gray-900);   /* dark surface */
  --color-text: var(--gray-50);       /* light text on dark */
}

Now names are always honest: --gray-50 is always light, --color-surface always means "the background." The semantic layer is the translation layer between "what colours exist" and "what colours are used where."

For multi-brand, it's different: When Brand B replaces Brand A, the semantic intent is identical — "primary" still means "primary." Each brand provides its own primitives file (with honest names: Brand A has --blue-500, Brand B has --coral-500) and maps them to the same semantic tokens (--color-primary). Alternatively, use neutral names like --brand-500 that don't claim a specific colour, so they can be swapped without the name becoming a lie.

Rule of thumb: Swap whatever layer lets all names remain truthful. If swapping a layer would make names lie about their values or their purpose, you're swapping the wrong layer.

Here are both dark mode approaches:

Approach 1: light-dark() inline (simple sites)

light-dark() is a native CSS colour function. You give it two values — the first is used in light mode, the second in dark mode:

color: light-dark(#1a1a2e, #e8e8e8);
/*                 ↑ light    ↑ dark  */

It decides which value to return based on the color-scheme property in effect on that element. The resolution chain:

  1. Explicit color-scheme on the element — color-scheme: dark forces dark
  2. Inherited from parent — if nothing is set on this element, it inherits
  3. OS preference — if nothing in the tree overrides it, the user's system setting (e.g. macOS Appearance) decides

To opt in to automatic light/dark based on the user's OS, declare this once:

:root {
  color-scheme: light dark;  /* "I support both — follow the OS" */
}

To let a toggle button override the OS preference, add/remove a class that forces color-scheme:

/* CSS */
:root { color-scheme: light dark; }       /* default: follow OS */
:root.dark  { color-scheme: dark; }        /* user forced dark */
:root.light { color-scheme: light; }       /* user forced light */

// JS — remove both, then add the one you want (or neither for "system")
const root = document.documentElement;
root.classList.remove('light', 'dark');
root.classList.add('dark');  // omit this line entirely to follow OS

The class names (.dark, .light) are arbitrary — you could call them anything. What matters is that the CSS rule they trigger sets color-scheme to the spec-defined keywords dark or light. Those are the values light-dark() reads from.

The single source of truth: color-scheme is the property that light-dark() reads from. Everything else — classes, JS toggles, media queries — is just a way to set that property. If you understand this, the whole mechanism is clear.

Here's a full token example using light-dark():

:root {
  color-scheme: light dark;

  /* Each semantic token declares both values inline */
  --color-surface: light-dark(#fefefe, #1a1a2e);
  --color-text: light-dark(#1a1a2e, #e8e8e8);
  --color-primary: light-dark(#3a86ff, #6eaaff);
  --color-border: light-dark(#e0e0e0, #333);
}

/* Toggle: override color-scheme on :root */
:root.dark { color-scheme: dark; }
:root.light { color-scheme: light; }

Approach 2: class-based semantic swap (multi-brand / complex apps)

/* Primitives stay the same regardless of theme */
:root {
  --blue-500: #3a86ff;
  --blue-300: #6eaaff;
  --gray-50: #fefefe;
  --gray-800: #1a1a2e;
  --gray-900: #0d0d1a;
}

/* Light semantics (default) */
:root, [data-theme="light"] {
  --color-surface: var(--gray-50);
  --color-text: var(--gray-800);
  --color-primary: var(--blue-500);
}

/* Dark semantics — same names, different primitives */
[data-theme="dark"] {
  --color-surface: var(--gray-900);
  --color-text: #e8e8e8;
  --color-primary: var(--blue-300);
}

When to use which:

• light-dark() is strictly binary — it only supports two modes (light and dark). There's no way to add a third value for high-contrast, sepia, or brand variants. It's elegant for simple light/dark toggling with few tokens. One declaration per token, no duplication.

• Class/attribute swap scales to any number of themes because each theme is just another block of custom property overrides. Need a third, fourth, fifth theme? Add another [data-theme="..."] block. This is what you use for multi-brand, high-contrast modes, or anything beyond the light/dark binary.

In practice, large apps use approach 2. light-dark() is best when dark mode is your only theming concern.

  --brand-text: #1a1a2e;
}

  --brand-primary: #e76f51;

  --brand-text: #2d1b14;

/* Component styles consume tokens — identical for both brands */
.brand-panel {

  color: var(--brand-text);
}

  background: var(--brand-primary);
  color: white;

/* In production, you'd use a data attribute instead of classes:
   [data-brand="fintech"] { --brand-primary: #3a86ff; ... }
   [data-brand="food"]    { --brand-primary: #e76f51; ... }
   Then set data-brand on a high-level container (or :root). */

The architectural insight: In a multi-brand system, you're not writing different CSS per brand. You're writing one set of component CSS that consumes semantic tokens. The brand is just a different set of token values injected at a high point in the DOM. This is the same principle as Angular's dependency injection — components declare what they need, the provider (theme) supplies it.

Scoped theming with data attributes and inheritance

Not everything on a page needs to be the same theme. Marketing panels, embedded widgets, or "dark hero" sections might override the page theme locally. This works via custom property inheritance — no @scope needed — because setting data-theme on a container causes the token overrides to cascade into all descendants:

Two cards on the same page, each with a scoped theme override

Light region

Default page theme. Tokens inherit from :root.

Dark region

Scoped override — just set data-theme="dark" on this container.

<!-- HTML — two regions on the same page with different themes -->
<div class="layout" data-theme="light">
  <div class="card">
    <h4>Light region</h4>
    <p>Default page theme. Tokens inherit from the parent.</p>
  </div>

  <div class="card" data-theme="dark">
    <h4>Dark region</h4>
    <p>Scoped override — just set data-theme="dark" on this container.</p>
  </div>
</div>

/* The same semantic blocks from before — they apply wherever
   data-theme appears in the DOM, at any nesting level */
[data-theme="light"] {
  --color-surface: #fefefe;
  --color-text: #1a1a2e;
  --color-primary: #3a86ff;
}
[data-theme="dark"] {
  --color-surface: #1a1a2e;
  --color-text: #e8e8e8;
  --color-primary: #6eaaff;
}

/* Components don't know or care which theme they're in */
.card { background: var(--color-surface); color: var(--color-text); }

Why this works: Custom properties inherit down the DOM tree. When you set data-theme="dark" on a container, the semantic token overrides cascade into every descendant. Components inside that container receive the dark values without knowing they're in a dark region. It's the CSS version of a scoped provider in Angular's DI system.

When @scope adds value to theming

The data-theme + inheritance pattern handles token values well. But what if you also want different structural styles (not just different colours) within a theme region? That's where @scope from Lesson 19 adds value — it lets you write rules that only apply inside a theme boundary, without leaking outward or inward past a limit:

@scope with a lower boundary — dark link styles stop at the nested light widget

Dark hero

Nested light widget:

Widget link ← NOT underlined (lower boundary stopped it) 
<!-- HTML -->
<main data-theme="light">
  <section data-theme="dark" class="hero">
    <h1>Dark hero</h1>
    <a href="#" class="link">Learn more</a>
    <div class="nested-widget" data-theme="light">
      <!-- This nested light region should NOT get dark link styles -->
      <a href="#" class="link">Widget link</a>
    </div>
  </section>
</main>

/* Tokens via inheritance (same as before) */
[data-theme="dark"] {
  --color-surface: #1a1a2e;
  --color-text: #e8e8e8;
  --color-primary: #6eaaff;
}

/* Structural/behavioural styles SCOPED to dark regions only,
   with a lower boundary so they don't leak into nested light regions */
@scope ([data-theme="dark"]) to ([data-theme="light"]) {
  .link {
    /* Underline style only for links inside dark regions */
    text-decoration: underline;
    text-underline-offset: 3px;
    /* This rule does NOT apply to .link inside the nested light widget */
  }

  h1 {
    /* Larger text shadow only makes sense on dark backgrounds */
    text-shadow: 0 2px 8px rgba(0, 0, 0, 0.5);
  }
}

Inheritance vs @scope — different jobs:

• Custom property inheritance handles token values (colours, spacing). It flows naturally down the tree and is overridden at any point by setting new values.

• @scope handles structural styles (layout changes, decorations, behaviours) that should only apply within a theme boundary and stop at a lower boundary. The to clause prevents leaking into nested counter-themes.

For most theming, inheritance alone is enough. Add @scope when you have theme-dependent styles that go beyond token values — e.g. dark regions need different link underlines, shadow treatments, or layout adjustments that shouldn't bleed into re-themed children.

Angular integration patterns

In Angular, the token architecture maps cleanly to several patterns:

Pattern 1: Theme service + host binding

// theme.service.ts
@Injectable({ providedIn: 'root' })
export class ThemeService {
  private theme = signal<'light' | 'dark' | 'system'>('system');

  readonly resolved = computed(() => {
    const t = this.theme();
    if (t !== 'system') return t;
    return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  });

  set(theme: 'light' | 'dark' | 'system') { this.theme.set(theme); }
}

// app.component.ts
@Component({
  selector: 'app-root',
  host: { '[attr.data-theme]': 'theme.resolved()' },
  template: `<router-outlet />`
})
export class AppComponent {
  theme = inject(ThemeService);
}

Pattern 2: Multi-brand via environment or route data

// Brand tokens live in a CSS file per brand, loaded conditionally
// styles/brand-fintech.css
:root {
  --brand-primary: #3a86ff;
  --brand-surface: #f0f7ff;
}

// styles/brand-food.css
:root {
  --brand-primary: #e76f51;
  --brand-surface: #fef5f2;
}

// angular.json — each build configuration includes a different brand file
"configurations": {
  "fintech": { "styles": ["src/styles/tokens.css", "src/styles/brand-fintech.css"] },
  "food":    { "styles": ["src/styles/tokens.css", "src/styles/brand-food.css"] }
}

Pattern 3: Scoped theme directive

// theme-scope.directive.ts — apply a theme to any container
@Directive({
  selector: '[appThemeScope]',
  host: { '[attr.data-theme]': 'appThemeScope()' }
})
export class ThemeScopeDirective {
  appThemeScope = input.required<'light' | 'dark'>();
}

<!-- Usage: dark hero on an otherwise light page -->
<section appThemeScope="dark" class="hero">
  <app-cta-button />  <!-- inherits dark tokens automatically -->
</section>

The principle: Angular components should never import theme values directly. They consume CSS custom properties, and the theme is provided by a data attribute set at the app shell or section level. This keeps component CSS portable across themes and brands.

Think of it as: tokens are to CSS what DI providers are to Angular — the consuming code declares dependencies, the context supplies values.

System preference + user override + persistence

The complete toggle pattern combines three concerns:

/* 1. Respect system preference by default */
:root {
  color-scheme: light dark;
  --color-surface: light-dark(#fefefe, #1a1a2e);
  --color-text: light-dark(#1a1a2e, #e8e8e8);
}

/* 2. User override via class on :root */
:root.light { color-scheme: light; }
:root.dark  { color-scheme: dark; }

/* 3. Persist in localStorage, restore on load */
// theme-toggle.ts — the JS side
const STORAGE_KEY = 'theme-preference';

function getTheme(): 'light' | 'dark' | 'system' {
  return (localStorage.getItem(STORAGE_KEY) as any) ?? 'system';
}

function applyTheme(theme: 'light' | 'dark' | 'system') {
  const root = document.documentElement;
  root.classList.remove('light', 'dark');
  if (theme !== 'system') root.classList.add(theme);
  localStorage.setItem(STORAGE_KEY, theme);
}

// Apply on page load (put in <head> to avoid flash)
applyTheme(getTheme());

Flash of wrong theme: The biggest UX pitfall. If you apply the theme in a deferred script, the page renders with the wrong theme first, then flips. Fix: put the theme restoration script in <head> as a blocking script (tiny, fast) — it runs before first paint. In Angular SSR, render the data-theme attribute server-side based on a cookie.

Token naming conventions

A naming system that scales:

/* Pattern: --{category}-{property}-{variant}-{state} */

/* Colours */
--color-primary                /* default primary */
--color-primary-hover          /* primary on hover */
--color-surface                /* page/card background */
--color-surface-raised         /* elevated surface (cards, modals) */
--color-text                   /* default body text */
--color-text-muted             /* secondary/disabled text */
--color-border                 /* default borders */
--color-danger                 /* errors, destructive actions */
--color-success                /* confirmations */

/* Spacing */
--space-1 through --space-8    /* primitives (scale) */
--space-inline                 /* horizontal padding */
--space-block                  /* vertical padding */
--space-gap                    /* flex/grid gaps */

/* Typography */
--font-body                    /* body font stack */
--font-heading                 /* heading font stack */
--font-mono                    /* code font stack */
--font-size-sm / -md / -lg     /* scale steps */
--line-height-tight / -normal  /* leading */

/* Radii, shadows, motion */
--radius-sm / -md / -lg / -full
--shadow-sm / -md / -lg
--duration-fast / -normal / -slow
--easing-default

Naming principle: Semantic tokens should describe purpose, not appearance. --color-primary not --color-blue. --color-danger not --color-red. When you switch brands, "blue" might become coral, but "primary" always means primary.

OKLCH + relative colour syntax in token architecture

In Lesson 18 you learned oklch and relative colour syntax. Here's where it becomes powerful at scale: you define a single base hue per brand, and the entire palette — hover states, muted variants, surfaces, dark mode — is computed from that one token.

The idea: one token, entire palette derived

/* Brand defines ONE base colour in oklch */
:root {
  --brand-hue: 250;         /* just the hue angle */
  --brand-chroma: 0.15;     /* saturation intensity */

  /* The single source-of-truth token */
  --brand-base: oklch(0.55 var(--brand-chroma) var(--brand-hue));
}

/* Derive the ENTIRE semantic palette via relative colour syntax */
:root {
  /* Primary shades — adjust lightness */
  --color-primary:       var(--brand-base);
  --color-primary-hover: oklch(from var(--brand-base) calc(l - 0.08) c h);
  --color-primary-light: oklch(from var(--brand-base) 0.92 0.04 h);

  /* Surfaces — same hue, very high lightness, low chroma */
  --color-surface:       oklch(from var(--brand-base) 0.985 0.005 h);
  --color-surface-raised:oklch(from var(--brand-base) 0.96 0.01 h);

  /* Text — same hue family, very low lightness */
  --color-text:          oklch(from var(--brand-base) 0.15 0.02 h);
  --color-text-muted:    oklch(from var(--brand-base) 0.45 0.02 h);

  /* Borders — low chroma, mid lightness */
  --color-border:        oklch(from var(--brand-base) 0.85 0.01 h);

  /* Danger — shift hue to red region, keep brand's chroma */
  --color-danger:        oklch(from var(--brand-base) l c 25);
  --color-success:       oklch(from var(--brand-base) l c 145);
}

Why this is powerful: To create Brand B, you change two numbers (hue and chroma). Every derived token recomputes. No manual colour picking for each variant. The relationships between colours are preserved across brands because they're defined as mathematical operations, not hard-coded values.

How you swap the brand base

The --brand-hue and --brand-chroma values are just custom properties on :root. You override them the same way you override any token — via a data attribute, a class, or a separate brand file:

/* Default brand (set on :root) */
:root {
  --brand-hue: 250;
  --brand-chroma: 0.15;
  --brand-base: oklch(0.55 var(--brand-chroma) var(--brand-hue));
}

/* Brand B: override hue and chroma on a data attribute */
[data-brand="food"] {
  --brand-hue: 25;
  --brand-chroma: 0.18;
  --brand-base: oklch(0.6 var(--brand-chroma) var(--brand-hue));
}

/* All the semantic tokens (--color-primary, --color-surface, etc.)
   use oklch(from var(--brand-base) ...) — they recompute automatically
   when --brand-base changes. No other CSS needs to change. */

The mechanism is the same as every other theme swap in this lesson:

<!-- Set it on <html> for the whole page -->
<html data-brand="food">

<!-- Or set it on a section for a scoped brand region -->
<section data-brand="food">...</section>

// JS — swap brand at runtime
document.documentElement.dataset.brand = 'food';

// Angular — host binding on AppComponent
@Component({
  host: { '[attr.data-brand]': 'brandService.current()' }
})

One attribute change at the root, and the entire palette recomputes via the oklch derivation chain.

Dark mode via lightness inversion

Since oklch separates lightness from hue and chroma, dark mode becomes a lightness transformation:

/* Light mode (default) — high lightness surfaces, low lightness text */
:root, [data-theme="light"] {
  --color-surface:    oklch(from var(--brand-base) 0.985 0.005 h);
  --color-text:       oklch(from var(--brand-base) 0.15 0.02 h);
  --color-primary:    oklch(from var(--brand-base) 0.55 c h);
}

/* Dark mode — flip the lightness relationship */
[data-theme="dark"] {
  --color-surface:    oklch(from var(--brand-base) 0.12 0.01 h);
  --color-text:       oklch(from var(--brand-base) 0.9 0.02 h);
  --color-primary:    oklch(from var(--brand-base) 0.72 c h);  /* lighter primary for contrast */
}

The mental model: In light mode, surfaces are near L=1.0 and text is near L=0.15. In dark mode, surfaces are near L=0.12 and text is near L=0.9. You're flipping the lightness axis while keeping hue and chroma stable. The brand identity (hue) is preserved across both modes — only the luminance relationship inverts.

Multi-brand with computed palettes

/* Brand A: cool blue fintech */
[data-brand="fintech"] {
  --brand-hue: 250;
  --brand-chroma: 0.15;
  --brand-base: oklch(0.55 var(--brand-chroma) var(--brand-hue));
}

/* Brand B: warm coral food delivery */
[data-brand="food"] {
  --brand-hue: 25;
  --brand-chroma: 0.18;
  --brand-base: oklch(0.6 var(--brand-chroma) var(--brand-hue));
}

/* The semantic layer is IDENTICAL for both brands — it just reads --brand-base */
/* All the oklch(from var(--brand-base) ...) tokens recompute automatically */

This is the ultimate payoff of oklch in a token system: brands become parameters, not separate stylesheets. The palette logic is written once and produces perceptually consistent results regardless of the input hue.

Putting it all together: the file structure

styles/
├── tokens/
│   ├── primitives.css       /* raw palette, spacing scale, radii */
│   ├── semantic-light.css   /* semantic tokens for light theme */
│   ├── semantic-dark.css    /* semantic tokens for dark theme */
│   └── components.css       /* per-component token overrides (optional) */
├── brands/
│   ├── brand-default.css    /* primitive overrides for default brand */
│   ├── brand-fintech.css    /* primitive overrides for fintech brand */
│   └── brand-food.css       /* primitive overrides for food brand */
└── global.css               /* resets, base typography, utilities */

In Angular, import them in angular.json styles array. For multi-brand, use build configurations to swap the brand file. For single-brand with light/dark, include both semantic files and switch via data-theme or color-scheme.
