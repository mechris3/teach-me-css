Lesson 10

Container Queries — Responsive to Parent, Not Viewport

Media queries ask: "How wide is the viewport?" Container queries ask: "How wide is the parent?" This lets components adapt to the space they're given — whether that's a full-width main area or a narrow sidebar — without knowing anything about the viewport.

The problem container queries solve

With media queries, a card component has to guess: "If the viewport is small, I'm probably in a narrow space." But that breaks when the same card appears in a sidebar (narrow) on a wide screen. The viewport is large, but the available space is small. Media queries can't handle this.

Mental model: Media queries are about the screen. Container queries are about the component's context. A component should respond to how much space it has, not what device it's on.

Two steps: declare a container, then query it

Container queries require two things:

  1. Mark an element as a container with container-type
  2. Write a @container rule that applies styles when the container meets a condition

Drag the right edge to resize — the card layout changes based on container width

↔ drag to resize

Card Title
This card changes from vertical to horizontal layout when its container is wide enough. No media queries involved.
/* 1. Declare the container */
.card-wrapper {
  container-type: inline-size;
  container-name: card-container;
}

/* 2. Query it */
@container card-container (min-width: 350px) {
  .card {
    flex-direction: row;  /* switch from stacked to horizontal */
  }
}

@container card-container (min-width: 500px) {
  .card {
    padding: 1.5rem;  /* more breathing room when space allows */
  }
}

container-type — what can be queried

The container-type property tells the browser what dimensions to track:

  • inline-size — track the inline axis width (horizontal in LTR). This is what you almost always want.
  • size — track both inline and block (width AND height). Rarely needed and has performance implications.
  • normal (default) — not a query container. Can still be queried for style queries (see below).

Why not just size? Tracking height requires the browser to know the element's height before laying out its children — which creates a circular dependency (children determine height, but height determines children's styles). inline-size avoids this because width is determined by the container's parent, not its children.

container-name — targeting specific containers

If you have nested containers, naming lets you target a specific one:

The card queries "card-container" specifically, not any ancestor container

↔ drag to resize

Named Query
This targets container-name: card-container specifically.
.wrapper {
  container-type: inline-size;
  container-name: card-container;
}

/* Targets only "card-container", ignores other ancestors */
@container card-container (min-width: 400px) {
  .card { flex-direction: row; }
}

/* Without a name — queries the nearest ancestor container */
@container (min-width: 400px) {
  .card { flex-direction: row; }
}

The shorthand container combines both:

container: card-container / inline-size;
/* equivalent to: container-name: card-container; container-type: inline-size; */

Container query units — relative to the container

Just like viewport units (vw, vh), container queries introduce units relative to the container:

  • cqi — 1% of the container's inline size (width)
  • cqb — 1% of the container's block size (height)
  • cqw — 1% of container width (same as cqi in horizontal writing)
  • cqh — 1% of container height
  • cqmin — the smaller of cqi and cqb
  • cqmax — the larger of cqi and cqb

Font size uses cqi — resize the container and the text scales with it

↔ drag to resize

This text scales with the container width using font-size: clamp(0.8rem, 3cqi, 1.5rem)
.container {
  container-type: inline-size;
}

.text {
  /* Fluid font size relative to container, not viewport */
  font-size: clamp(0.8rem, 3cqi, 1.5rem);

  /* Padding relative to container width */
  padding: 2cqi;
}

cqi vs vw: vw is always relative to the viewport. cqi is relative to the nearest container. A component using cqi scales correctly whether it's full-width or in a narrow sidebar — because it responds to its own container, not the screen.

Style queries — querying custom property values

Beyond size, you can query the computed style of a container — specifically custom property values. This lets you create style-based variants without extra classes:

Click the button to toggle --theme: the card responds via a style query

I respond to @container style(--theme: dark). No class on me — just the container's custom property value.
.container {
  container-type: normal;  /* style queries don't need inline-size */
  --theme: light;  /* this IS a custom property declaration — same as Lesson 1 */
}

.container.dark { --theme: dark; }

/* The child responds to the container's custom property */
@container style(--theme: dark) {
  .card {
    background: #1a1a2e;
    color: white;
  }
}

Style queries + custom properties = component variants without classes on the component itself. Set a custom property on an ancestor, and descendants respond. This is how design systems can propagate themes without threading classes through every component.

Browser support: Style queries for custom properties are supported in Chrome 111+, Safari 18+. Querying standard properties (like style(display: flex)) is specced but not implemented in any browser yet.

What about querying standard properties?

You might wonder: "Can I query @container style(display: flex)?" Not yet — no browser supports it. Only custom properties work today.

Workaround: mirror a standard property into a custom property:

.container {
  display: flex;
  --display: flex;  /* manual mirror */
}

@container style(--display: flex) {
  .child { /* responds to the custom property flag */ }
}

The limitation: --display doesn't automatically stay in sync with display. If another rule changes display, --display goes stale unless you update both. It's a manual semantic signal, not a live reflection of computed style.

This pattern works well for things you control deliberately (--layout: sidebar, --density: compact) but isn't a substitute for true standard property queries. For "if this container happens to be flex, style children differently" — you'd need the not-yet-implemented feature, or use a class/attribute you already control.

Media queries vs Container queries — when to use which

Use media queries when... Use container queries when...
Page-level layout changes (sidebar collapses) Component-level layout changes (card stacks/unstacks)
Global font size / spacing scale shifts Component font/spacing adapts to available space
You need to respond to device capabilities (hover, pointer) Same component used in different width contexts
The viewport IS the relevant context The parent IS the relevant context

Browser support

Container size queries: Chrome 105+, Safari 16+, Firefox 110+. Fully safe for production.

Container style queries (custom properties): Chrome 111+, Safari 18+, Firefox 🚧 (in progress).

Container query units (cqi, etc.): Chrome 105+, Safari 16+, Firefox 110+.
